# ニコニコ生放送（PC）コメント一覧 DOM・周辺の調査メモ

外部調査（Compass 等）のアーティファクトを、nicolivelog の開発用にリポジトリへ取り込んだ要約です。**公式の一次情報ではありません**。規約の解釈・法的判断は各自で確認してください。

---

## 1. DOM 構造（ライブ / タイムシフト）

- **ライブとタイムシフトで同一**とされる。  
  `.table-row[data-comment-type="normal"]` と `.table-cell` 系、`[class*="___comment-data-grid___"]`（または実機では `comment-data-grid` / `ga-ns-comment-panel` など）のコンテナ。
- CSS Modules によりクラス名にハッシュが付くため、**安定クラス**（`table-row`, `comment-text`, `data-comment-type` 等）と **属性セレクタ**の併用が現実的。

→ 本拡張の [`src/lib/nicoliveDom.js`](../src/lib/nicoliveDom.js) の方針と整合。

---

## 2. URL

- 同一パス: `https://live.nicovideo.jp/watch/lv{番号}`
- タイムシフトでは **`#05:00` のようなハッシュ**で時間指定がある場合がある（フラグメントはサーバに送られないが、クライアント側の再生位置に利用）。

→ `extractLiveIdFromUrl` は pathname の `lv` を参照する想定で、**ハッシュの有無は通常 lv 抽出に影響しない**。

---

## 3. 仮想スクロール

- **ライブ**: 新着に合わせて一覧が動く想定。
- **タイムシフト**: **再生位置と一覧が同期**する想定。
- いずれも React 仮想リストの利用が強く推定される。

→ [`src/lib/commentHarvest.js`](../src/lib/commentHarvest.js) のスクロール走査は、タイムシフトでも「表示レンジに載るコメント」を拾う用途では有効になりうるが、**再生位置によっては DOM に載らない区間**があり得る。

---

## 4. 2024 年頃のコメント基盤変更（参考）

- コメント配信が JSON/WebSocket 中心から **Protocol Buffers / HTTP ストリーミング**寄りに変わった、という整理がある。
- 一方で **フロントのコメント一覧 DOM のパターンは概ね維持**されている、という整理。

→ 将来、内部 API に依存する実装は壊れやすい。**DOM ベースは壊れにくいが、UI 改修には弱い**というトレードオフは変わらない。

---

## 5. ユーザー ID

- ライブ・タイムシフトで **DOM 上の露出に大きな差はない**、という整理。
- 右クリックメニュー等から取得できる情報がある一方、**一覧行の DOM にはユーザー ID が載らない**ことが多い。

→ ポップアップの「ID未取得」まとまりは、この制約と一致。

---

## 6. 規約・コンプライアンス（一般論・要確認）

調査側の整理として、次が**論点として挙げられている**（最終判断は利用規約本文とご自身の用途で確認すること）。

- 利用規約上の **アドイン等に関する条項**（調査では「§2」等と言及）と、ブラウザ拡張の位置づけの解釈。
- **著作権・利用許諾**に関する条項（調査では「§4」等と言及）と、コメント本文の複製・蓄積。
- **公式 API が生放送コメントに対応していない**、という整理。
- 特許・実装リスクは **映像オーバーレイ表示**など別系統に集中しやすい、という整理（コメントテキスト抽出とは切り分け可能、という意味合い）。

**本リポジトリは法的助言を提供しません。**

---

## 7. nicolivelog での使い方

- §3 の「タイムシフトでも同じ DOM か」を外部 LLM に聞く前に、**本メモを添付**すると会話の前提が揃いやすい。
- 実機で DOM が変わったら、**DevTools でコメント行の HTML を取り**、[`nicoliveDom.js`](../src/lib/nicoliveDom.js) のセレクタを更新する。
